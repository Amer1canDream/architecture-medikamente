# Медикаменте — архитектура и защита данных

## Задание 1. Анализ безопасности (As-Is)

Диаграммы потоков данных (draw.io) в папке `Task1/`: регистрация и запись, приём и медкарта, лаборатория, оплата и бухгалтерия, учёт ТМЦ.

Основные проблемы:

- нет разграничения доступа — только доменная аутентификация, RBAC/ABAC отсутствует;
- нет аудита действий с файлами и записями;
- персональные данные и данные о здоровье хранятся без шифрования и меток чувствительности;
- данные пациента дублируются в разных Excel и папках без единого источника правды;
- передача данных по почте, ККМ и 1С не защищена и не логируется;
- нет процедуры удаления/анонимизации данных по запросу клиента (152-ФЗ);
- все сервисы на одном физическом сервере — единая точка отказа и компрометации.

Классификация данных:

| Данные | Класс | Необходимость защиты |
|---|---|---|
| ФИО, дата рождения, телефон, email | PII | прямая идентификация, риски по 152-ФЗ |
| Адрес, место работы, соцданные | PII (расширенные) | риск деанонимизации |
| Диагнозы, анализы, заключения | PHI | наивысшая чувствительность, вред пациенту при утечке |
| Платежи, договор | FINANCIAL | финансовый и репутационный риск |
| Сканы документов | PII/документ | подделка при утечке |
| Данные ТМЦ | INTERNAL | коммерческая информация |

## Задание 2. Целевая архитектура (To-Be)

Диаграмма контекста C4: `Task2/C4_Context_Medikamente_TOBE.drawio`.

Добавлены два новых слоя поверх единой платформы данных пациентов:

- **Privacy by Design Layer** — Consent & Access Management (согласия, RBAC/ABAC), Data Classification & Tagging (метки PII/PHI/FINANCIAL), Audit & Monitoring (журналирование, алерты). Все запросы к данным проходят через эти сервисы, а не проверяются точечно в каждой системе.
- **Аналитический слой (BI/ML/AI)** — Data Lake & Analytics Platform получает данные только после прохождения через Data Classification, поэтому в аналитику не попадают непомеченные PII/PHI.

## Задание 3. Шифрование данных at rest и in transit

Классификация по состоянию и способу защиты:

| Данные | Состояние | Необходимость защиты | Инструменты защиты |
|---|---|---|---|
| ФИО, контакты, адрес (PII) | at rest (БД, файлы) | утечка ведёт к деанонимизации и штрафам по 152-ФЗ | шифрование диска/БД (LUKS, TDE), pgcrypto для отдельных полей |
| Диагнозы, анализы, заключения (PHI) | at rest | высшая чувствительность, прямой вред пациенту | шифрование на уровне поля, ключи в KMS/HSM, доступ только через RBAC |
| Платёжные данные, договоры | at rest | финансовый и репутационный риск | шифрование БД 1С/платформы, ограниченный доступ по ролям |
| Запросы пациент → портал, портал → платформа | in transit | перехват трафика в локальной сети или интернете | TLS 1.2+ everywhere, mTLS между сервисами |
| Обмен с лабораторией и платёжным шлюзом (внешние API) | in transit | внешний канал, выше риск перехвата | TLS + подписанные API-контракты, VPN/IPsec до внешних партнёров |
| Резервные копии | at rest | долгий срок хранения повышает риск | шифрование бэкапов, отдельное хранение ключей |
| Логи и аудит-данные | at rest | могут содержать косвенные PII | шифрование, ограниченный доступ, ротация/удаление по сроку |

Автоматизированный контроль:

- SIEM (например, ELK/Wazuh) — сбор и корреляция логов доступа, алерты на аномалии;
- DLP для почты и файлового сервера — блокировка исходящей отправки PII/PHI;
- сканер конфигураций/секретов в CI — проверка, что новые релизы не хранят ключи и пароли в коде;
- мониторинг сертификатов и сроков действия ключей шифрования;
- интеграция всех перечисленных средств с Audit & Monitoring из Задания 2.

Реестр средств защиты:

| Средство | Тип | Назначение |
|---|---|---|
| LUKS / BitLocker | ПО | шифрование дисков файлового сервера и БД |
| pgcrypto / TDE (Postgres, 1С) | ПО | шифрование отдельных полей и БД |
| TLS 1.2+/1.3 (nginx, API Gateway) | ПО | шифрование трафика между сервисами и клиентами |
| HashiCorp Vault / KMS | ПО | управление и ротация ключей шифрования |
| HSM | аппаратное | хранение мастер-ключей |
| VPN/IPsec-шлюз | сеть | защищённый канал с лабораторией и филиалами |
| DLP-система | ПО | контроль исходящих писем и файлов |
| SIEM (ELK/Wazuh) | ПО | сбор логов, обнаружение аномалий, алертинг |
| WAF / API Gateway с mTLS | ПО | защита внешних API-интеграций |

## Задание 4. Узкие места при миграции

Диаграмма Исикавы: `Task4/Ishikawa_Migraciya.drawio`. Следствие — узкие места и риски производительности при миграции на микросервисную архитектуру. Категории причин:

- **Инфраструктура** — один физический сервер без резервирования, нет сегментации сети, ограниченные каналы под филиалы, нет автомасштабирования.
- **Архитектура и ПО** — 1С в файловом режиме плохо режется на сервисы, нет API-контрактов, разнородный стек без единой модели данных, нет CI/CD.
- **Данные** — дублирование данных пациента в Excel, нет единой схемы и справочников, много неструктурированных сканов, риск потери целостности при переносе.
- **Интеграция** — ККМ/1С на устаревшем OLE, у лаборатории и партнёров нет API-контрактов, синхронные интеграции как точка отказа, нет очередей для асинхронного обмена.
- **Персонал** — 3 специалиста в IT на фоне 5-кратного роста, технический директор один совмещает роль архитектора, привычка к ручным процессам, нет опыта контейнеризации legacy 1С.
- **Процессы** — нет формализованных бизнес-процессов, нет управления изменениями и релизами, нет плана отката, не продуман dual-run старой и новой системы.

Рекомендации (по приоритету):

1. Высокий — вынести данные пациентов из Excel в единую БД до начала декомпозиции на сервисы; описать API-контракты для 1С, ККМ и лаборатории; подготовить план отката и dual-run.
2. Средний — нанять/обучить инженеров под микросервисы и контейнеризацию, ввести CI/CD, внедрить очередь сообщений для асинхронных интеграций.
3. Низкий — сегментация сети и резервирование инфраструктуры, формализация процессов управления изменениями — можно делать параллельно с миграцией.

## Задание 5. Cutover-план миграции

**Выбранная стратегия — Parallel Run.** Компания работает с медицинскими данными в реальном времени (запись, приём, оплата), простой недопустим, а High-level design и Branch by abstraction предполагают работу с существующим кодом/сервисами, которого по сути нет — сейчас это ручной Excel/1С процесс, а не система для рефакторинга слоями. Parallel run позволяет вести старую и новую систему одновременно, сверять данные и переводить отделы по одному, откатываясь без потери данных, если что-то идёт не так.

Ограничение подхода — двойной ввод данных нагружает персонал и требует времени на синхронизацию, поэтому период parallel run нужно держать как можно короче для каждого отдела.

Cutover-план:

| Этап | Описание действия | Ключевые задачи | Ответственные | Время/Сроки | Риски и меры по снижению |
|---|---|---|---|---|---|
| Подготовка | Перенос исторических данных в новую платформу, настройка окружения, обучение персонала | Миграция и сверка данных, настройка ролей RBAC/ABAC, обучение ресепшена | Технический директор, бизнес-аналитик | 2–3 недели | Потеря данных при переносе — контрольная сверка выгрузок до старта |
| Пилот на одном отделе (ресепшн) | Ресепшн работает в старой и новой системе параллельно | Двойной ввод, ежедневная сверка записей, сбор обратной связи | Технический директор, старший ресепшена | 1–2 недели | Отказ персонала от новой системы — ежедневный контроль факта использования |
| Синхронизация и валидация | Сверка данных старой и новой системы, устранение расхождений | Автоматическая сверка ID пациентов, фиксация и разбор расхождений | Бизнес-аналитик, DPO | 1 неделя после пилота | Рассинхронизация ПДн — единый ID пациента и лог изменений в обеих системах |
| Поэтапный перевод отделов | Перевод врачей, кассы/бухгалтерии, склада по одному отделу за раз | Старая система переходит в режим read-only для переведённых отделов | Технический директор, руководители отделов | 4–6 недель | Рост нагрузки при подключении нового отдела — проверка производительности перед стартом каждого этапа |
| Отключение старой системы | Перевод старой системы в архивный read-only режим | Финальная сверка, фиксация архивной копии | Технический директор | 1 неделя после последнего отдела | Ошибочное отключение до готовности — критерий: 2 недели новой системы без критичных инцидентов |
| Мониторинг после cutover | Наблюдение за производительностью, аудитом доступа и инцидентами | Контроль метрик из Audit & Monitoring (Задание 2) | Администратор сети | 30 дней после cutover | Скрытые ошибки миграции — усиленный мониторинг первый месяц, план отката из архива при критичном сбое |

План отката: на любом этапе до полного отключения старой системы отдел может вернуться на неё, так как она продолжает работать параллельно и не отключается до подтверждения стабильности. После отключения — восстановление из архива старой системы (read-only копия) как крайняя мера.

Роли: технический директор — владелец миграции и решений по переходу между этапами; бизнес-аналитик — валидация данных и отчётности; администратор сети — инфраструктура, бэкапы, мониторинг; DPO — контроль соответствия 152-ФЗ при переносе ПДн; руководители отделов — приёмочное тестирование и обратная связь от сотрудников.

## Задание 6. Движок классификации данных

Диаграмма C2: `Task6/C2_Data_Classification_Engine.drawio`.

Движок стоит перед аналитическим хранилищем и решает, что источники данных не размечены и их схема может меняться:

- **Ingestion Connector** принимает батчи из источников (Excel, 1С, сканы, API лаборатории).
- **Schema Discovery & Drift Detector** профилирует каждую партию и фиксирует изменения структуры, чтобы правила классификации не ломались молча при новом поле.
- **Rule-based Classifier** — быстрые правила/словари для известных полей (ФИО, телефон, ИНН и т.п.), **ML/NLP Classifier** — для свободного текста и сканов, где паттернов недостаточно.
- Записи с низкой уверенностью классификации уходят в **Human Review Queue**, их подтверждает бизнес-аналитик или DPO — так система не пропускает новые типы PII/PHI молча.
- **Tag & Metadata Catalog** хранит итоговые метки и версии схем, **Policy & Anonymization Module** применяет к данным правила доступа и обезличивания на основе тегов (используя Consent & Access Management из Задания 2), **Loader** пишет результат в хранилище. Все компоненты логируют решения в Audit & Monitoring.

Слои хранилища:

- **Raw/Landing zone** — данные как есть, зашифрованы, доступ deny-by-default до момента классификации.
- **Classified zone** — данные с проставленными тегами PII/PHI/FINANCIAL, доступ по RBAC/ABAC согласно тегу.
- **Curated/Anonymized zone** — обезличенные и агрегированные данные для BI/ML/AI, наполняется только из classified zone через Policy & Anonymization Module.

Дополнительные меры: карантин для записей с неуверенной классификацией (не публикуются в classified/curated до подтверждения), политики хранения и удаления по тегу, версионирование схем в каталоге меток.

Метрики эффективности классификации:

- покрытие — доля полей/файлов с проставленным тегом;
- precision/recall классификатора на контрольной выборке, размеченной вручную;
- false negative rate по PII/PHI — самая критичная метрика, отражает риск утечки в curated zone;
- доля записей в очереди ручной проверки — рост говорит о дрейфе схемы или новом типе данных, которого не покрывают правила;
- время от загрузки партии до появления в classified zone.

Эти метрики используются для оптимизации: рост доли ручной проверки — сигнал добавить правило или дообучить модель; рост false negative — повод ужесточить пороги уверенности перед публикацией в curated zone.

Масштабируемость: инкрементальная классификация только новых/изменённых записей вместо полного пересчёта; дешёвые rule-based правила разбирают основной объём, ML используется только там, где правила не сработали; ingestion и классификаторы масштабируются горизонтально по партициям (источник/дата); schema registry с версионированием позволяет быстро адаптировать правила при структурных изменениях источников без остановки пайплайна.
